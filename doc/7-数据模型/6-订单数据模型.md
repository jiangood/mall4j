# 订单数据模型

订单主表：

```text
tz_order
```

订单项表：

```text
tz_order_item
```

订单结算表：

```text
tz_order_settlement
```

退款表：

```text
tz_order_refund
```

## 订单主表核心字段

| 字段 | 说明 |
| --- | --- |
| `order_id` | 订单 ID。 |
| `shop_id` | 店铺 ID。 |
| `prod_name` | 订单商品名称，多个商品用逗号拼接。 |
| `user_id` | 下单用户 ID。 |
| `order_number` | 订单流水号。 |
| `total` | 订单总金额。 |
| `actual_total` | 实付金额。 |
| `pay_type` | 支付方式。 |
| `remarks` | 订单备注。 |
| `status` | 订单状态。 |
| `freight_amount` | 运费。 |
| `addr_order_id` | 订单收货地址 ID。 |
| `product_nums` | 商品总数。 |
| `is_payed` | 是否已支付。 |
| `delete_status` | 用户侧删除状态。 |
| `refund_sts` | 退款状态。 |
| `reduce_amount` | 优惠金额。 |

`order_number` 有唯一索引，是订单业务查询的关键字段。

## 订单项表核心字段

| 字段 | 说明 |
| --- | --- |
| `order_item_id` | 订单项 ID。 |
| `shop_id` | 店铺 ID。 |
| `order_number` | 订单号。 |

| `sku_id` | SKU ID。 |
| `prod_count` | 购买数量。 |
| `prod_name` | 商品名称。 |
| `sku_name` | SKU 名称。 |
| `pic` | 商品图片。 |
| `price` | 下单价格。 |
| `user_id` | 用户 ID。 |
| `product_total_amount` | 商品项总金额。 |
| `comm_sts` | 评价状态。 |

## 订单状态

| 值 | 含义 |
| --- | --- |
| `1` | 待付款 |
| `2` | 待发货 |
| `3` | 待收货 |
| `4` | 待评价 |
| `5` | 成功 |
| `6` | 失败或关闭 |

## 关联关系

```text
tz_order.order_number = tz_order_item.order_number
tz_order.order_number = tz_order_settlement.order_number
tz_order.addr_order_id = tz_user_addr_order.addr_order_id
```

订单地址会复制到订单地址表，避免用户后续修改收货地址影响历史订单。

## 完成后应该知道

- `tz_order` 是店铺维度订单主记录。
- `tz_order_item` 保存订单内具体商品项。
- `tz_order_settlement` 保存支付流水、应付金额和支付状态。
- `tz_user_addr_order` 保存下单时的地址快照。

## 检查题-含答案

问题：支付接口应该信前端传来的金额吗？

答案：不应该。当前支付流程按订单号读取 `tz_order_settlement` 中的金额，不接受前端传入金额作为支付依据。

问题：为什么一个用户一次提交可能生成多个订单？

答案：提交订单时按店铺拆单，每个店铺生成一个订单，便于商家发货、结算和确认收货。

问题：订单项和订单主表为什么用 `order_number` 关联？

答案：`order_number` 是订单业务流水号，订单项、结算表都围绕它关联订单业务数据。
