# 二开建表建议

新增后台管理功能时，表结构要尽量跟项目现有习惯一致。不要只为了当前页面能用就随便建字段。

## 先回答三个问题

| 问题 | 判断方式 |
| --- | --- |
| 这张表是否属于某个店铺？ | 属于店铺就加 `shop_id`。 |
| 这条数据是否需要软删除或上下架？ | 需要就加 `status` 或复用现有删除状态风格。 |
| 后台是否需要排序？ | 需要就加 `seq`。 |

## 常用字段

| 字段 | 什么时候加 |
| --- | --- |
| `shop_id` | 店铺隔离数据，例如公告、商品、分组。 |
| `status` | 启用/禁用、正常/删除、上下架。 |
| `seq` | 后台人工排序。 |
| `create_time` | 需要记录创建时间。 |
| `update_time` | 需要记录修改时间。 |
| `delete_time` | 软删除且需要记录删除时间。 |

## 帮助文章表示例

```sql
CREATE TABLE `tz_help_article` (
  `id` bigint NOT NULL AUTO_INCREMENT COMMENT '帮助文章ID',
  `shop_id` bigint DEFAULT NULL COMMENT '店铺ID',
  `title` varchar(100) NOT NULL COMMENT '标题',
  `category` varchar(50) DEFAULT NULL COMMENT '分类',
  `content` text COMMENT '内容',
  `status` tinyint DEFAULT 1 COMMENT '状态 1启用 0禁用',
  `seq` int DEFAULT 0 COMMENT '排序',
  `create_time` datetime DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='帮助中心文章';
```

## 建表后要同步的地方

- 实体 Model。
- Mapper 接口和 XML。
- Service 和 ServiceImpl。
- Controller。
- 前端 CRUD 字段配置。
- 菜单和按钮权限。
- 如果是初始化功能，还要把表结构和初始数据写入初始化 SQL 或迁移脚本。

## 检查题-含答案

问题：帮助中心文章是否应该加 `shop_id`？

答案：如果帮助文章是店铺维度管理，就应该加；如果是平台全局帮助中心，可以不加。当前示例按后台店铺隔离思路加 `shop_id`。

问题：`status` 字段能不能随便定义含义？

答案：不建议。应优先沿用项目已有习惯，例如 `1` 正常/启用，`0` 禁用/删除，避免前后端理解不一致。

问题：建完表就算二开完成了吗？

答案：不算。还要补后端、前端、菜单权限、角色授权和本地验证。
