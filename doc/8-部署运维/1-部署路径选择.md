# 部署路径选择

部署前先选路径。不要一上来就执行 `docker compose up -d`，因为当前仓库的 Docker Compose 还有需要修正的地方。

## 推荐优先级

| 场景 | 推荐方式 | 原因 |
| --- | --- | --- |
| 第一次上线验证 | jar + 外部 MySQL/Redis + Nginx | 最容易定位问题。 |
| 已熟悉 Docker | 修正 compose 后 Docker 部署 | 需要先处理 `db/Dockerfile` 缺失。 |
| 本地开发 | IDE/Maven + 本地 MySQL/Redis + Vite | 调试最快。 |

## jar 部署主线

适合先跑通生产环境：

1. 准备 MySQL，导入 `db/yami_shop.sql`。
2. 准备 Redis。
3. 修改 `application-prod.yml`。
4. 执行 `mvn clean package -DskipTests`。
5. 用 `prod` profile 启动两个 jar。
6. 构建 `front-end/mall4v`。
7. 用 Nginx 托管前端并转发接口。

## Docker 部署主线

当前不能直接无脑启动。需要先处理：

```text
docker-compose.yml 引用了 db/Dockerfile
当前 db/ 目录只有 yami_shop.sql
```

处理后再考虑：

1. 打包后端 jar。
2. 修正 MySQL 镜像或挂载初始化 SQL。
3. 确认 Docker 环境变量。
4. 启动容器。
5. 验证 `mall4j-admin`、`mall4j-api`、MySQL、Redis 都能互相访问。

## 完成后应该知道

- 新手先选 jar 部署更稳。
- Docker 不是不能用，但当前 compose 需要先修。
- Nginx 主要负责前端静态文件和接口反向代理。
- 生产端口要按实际 profile 配置确认，不要只看本地开发端口。

## 检查题-含答案

问题：为什么第一次上线不建议直接从 Docker Compose 开始？

答案：当前 `docker-compose.yml` 引用了缺失的 `db/Dockerfile`，直接执行会失败。先用 jar 跑通生产链路，更容易定位数据库、Redis、后端、前端、Nginx 的问题。

问题：jar 部署时，前端也要放进 jar 吗？

答案：不用。管理后台是前后端分离项目，前端用 `pnpm run build` 构建后交给 Nginx 托管。

问题：生产端口一定是 `8085` 和 `8086` 吗？

答案：不一定。本地和 Docker 配置常见是 `8085`、`8086`；当前 `application-prod.yml` 中管理端是 `8111`，用户端是 `8112`。
