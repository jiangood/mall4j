# 新增菜单与权限

这一篇只关注菜单、按钮和角色授权。目标是让页面能出现，按钮能按权限显示，接口能按权限拦截。

后台菜单和权限都来自 `tz_sys_menu`。

## 菜单类型

| type | 用途 |
| --- | --- |
| `0` | 目录，用于左侧一级或分组导航。 |
| `1` | 菜单，对应一个前端页面。 |
| `2` | 按钮，对应新增、修改、删除等操作权限。 |

## 新增菜单

菜单类型为 `1` 时，重点填写：

```text
parent_id: 上级目录 ID
name: 页面名称
url: 前端页面路径
type: 1
order_num: 排序
```

例如帮助中心文章：

```text
url = shop/helpArticle
```

前端会加载：

```text
front-end/mall4v/src/views/modules/shop/helpArticle/index.vue
```

## 新增按钮权限

按钮类型为 `2` 时，重点填写：

```text
parent_id: 所属菜单 ID
name: 按钮名称
perms: 权限标识
type: 2
```

示例：

```text
shop:helpArticle:info,shop:helpArticle:page
shop:helpArticle:save
shop:helpArticle:update
shop:helpArticle:delete
```

## 分配角色

菜单和按钮建好后，还需要在角色里分配权限。否则用户登录后 `/sys/menu/nav` 不会返回这些菜单或按钮权限。

## 完成后应该看到

- 登录后左侧菜单出现新页面。
- `/sys/menu/nav` 返回新菜单。
- 有按钮权限的角色能看到新增、修改、删除按钮。
- 没有按钮权限的角色看不到对应按钮。
- 直接访问无权限接口会被拦截。

## 检查题-含答案

问题：只新增菜单，不新增按钮权限，会发生什么？

答案：用户可能能进入页面，但新增、修改、删除按钮不会按权限正常显示，接口权限也缺少对应授权依据。

问题：只写前端 `isAuth()`，不写后端 `@PreAuthorize`，安全吗？

答案：不安全。前端只能隐藏按钮，不能保护接口。后端必须用 `@PreAuthorize` 做真正的权限控制。

问题：改完权限后为什么建议重新登录？

答案：前端权限来自登录后 `/sys/menu/nav` 返回的数据并存入 `sessionStorage`，重新登录能重新加载最新菜单和按钮权限。
