# 新增接口

这一篇只关注后端管理接口。目标是让你先把接口跑通，再接前端。

新增后台接口时，优先参考已有 Controller，例如：

```text
yami-shop-admin/src/main/java/com/yami/shop/admin/controller/NoticeController.java
```

## 基本结构

```text
@RestController
@RequestMapping("/shop/helpArticle")
public class HelpArticleController {
}
```

常见接口：

```text
GET    /page
GET    /info/{id}
POST
PUT
DELETE /{id}
```

## 返回格式

保持现有风格：

```text
ServerResponseEntity.success(...)
ServerResponseEntity.showFailMsg(...)
```

## 权限注解

需要权限控制的接口加：

```java
@PreAuthorize("@pms.hasPermission('shop:helpArticle:save')")
```

权限标识要和 `tz_sys_menu.perms`、前端 `isAuth()` 保持一致。

## 店铺数据

如果数据属于当前店铺，新增时要写入当前登录用户的 `shopId`：

```text
SecurityUtils.getSysUser().getShopId()
```

查询、修改、删除也应注意按 `shopId` 隔离，避免越权操作。

## 完成后应该看到

- 分页接口返回 `code=00000`。
- 新增接口能写入当前店铺数据。
- 修改接口只修改目标记录。
- 删除接口执行后，分页列表不再出现该记录。
- 未授权用户访问受控接口时返回未授权。

## 检查题-含答案

问题：后台管理接口为什么不放在 `yami-shop-api`？

答案：`yami-shop-api` 面向用户端、小程序、H5、uni-app；后台管理接口面向运营后台，应放在 `yami-shop-admin`。

问题：什么时候应该抛 `YamiShopBindException`？

答案：业务规则失败且需要给前端明确提示时，例如记录不存在、库存不足、状态不允许修改。

问题：查询、修改、删除为什么要考虑 `shopId`？

答案：后台数据通常按店铺隔离。只按主键操作可能越权影响其他店铺数据。
