# 新增后台管理功能-帮助中心文章

本示例用于说明如何新增一个完整后台管理功能。它不是要改造交易链路，而是给二开者一个低风险模板。

目标功能：在管理后台新增“帮助中心文章管理”，支持列表、新增、修改、删除、启用/禁用。

读完并照做后，你应该能独立完成一个后台 CRUD，并知道每一步完成后该如何验证。

## 学习方式

不要一次性写完所有文件。建议按下面四个小任务推进：

| 小任务 | 先做什么 | 完成后应该看到 |
| --- | --- | --- |
| 1 | 对照公告管理读一遍现有链路。 | 能找到 `NoticeController`、`shop/notice` 页面、`shop:notice:*` 权限。 |
| 2 | 完成帮助文章后端表、实体、Mapper、Service、Controller。 | 访问分页接口返回 `code=00000`。 |
| 3 | 完成帮助文章前端 CRUD 配置、列表页、表单页。 | 页面能展示列表，能打开新增和修改弹窗。 |
| 4 | 完成菜单、按钮权限和角色授权。 | 有权限用户能看到菜单和按钮，无权限用户看不到按钮。 |

## 表结构示例

```sql
CREATE TABLE `tz_help_article` (
  `id` bigint NOT NULL AUTO_INCREMENT COMMENT '帮助文章ID',
  `shop_id` bigint DEFAULT NULL COMMENT '店铺ID',
  `title` varchar(100) NOT NULL COMMENT '标题',
  `category` varchar(50) DEFAULT NULL COMMENT '分类',
  `content` text COMMENT '内容',
  `status` tinyint DEFAULT 1 COMMENT '状态 1启用 0禁用',
  `seq` int DEFAULT 0 COMMENT '排序',
  `create_time` datetime DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='帮助中心文章';
```

完成后应该能在数据库中看到 `tz_help_article` 表。

## 后端文件

建议按现有公告管理 `NoticeController` 的风格实现。

```text
yami-shop-bean/src/main/java/com/yami/shop/bean/model/HelpArticle.java
yami-shop-service/src/main/java/com/yami/shop/dao/HelpArticleMapper.java
yami-shop-service/src/main/java/com/yami/shop/service/HelpArticleService.java
yami-shop-service/src/main/java/com/yami/shop/service/impl/HelpArticleServiceImpl.java
yami-shop-service/src/main/resources/mapper/HelpArticleMapper.xml
yami-shop-admin/src/main/java/com/yami/shop/admin/controller/HelpArticleController.java
```

Controller 基本接口：

```text
GET    /shop/helpArticle/page
GET    /shop/helpArticle/info/{id}
POST   /shop/helpArticle
PUT    /shop/helpArticle
DELETE /shop/helpArticle/{id}
```

后端完成后，先不急着写前端。建议用接口文档或调试工具验证：

- `GET /shop/helpArticle/page` 能返回分页结构。
- `POST /shop/helpArticle` 能新增数据。
- `PUT /shop/helpArticle` 能修改数据。
- `DELETE /shop/helpArticle/{id}` 能删除数据。

权限标识：

```text
shop:helpArticle:page
shop:helpArticle:info
shop:helpArticle:save
shop:helpArticle:update
shop:helpArticle:delete
```

## 前端文件

```text
front-end/mall4v/src/crud/shop/helpArticle.js
front-end/mall4v/src/views/modules/shop/helpArticle/index.vue
front-end/mall4v/src/views/modules/shop/helpArticle/add-or-update.vue
```

页面请求路径要和后端保持一致：

```text
/shop/helpArticle/page
/shop/helpArticle/info/{id}
/shop/helpArticle
/shop/helpArticle/{id}
```

按钮权限要和后端一致：

```js
isAuth('shop:helpArticle:save')
isAuth('shop:helpArticle:update')
isAuth('shop:helpArticle:delete')
```

前端完成后，先用管理员账号验证页面，再用缺少按钮权限的角色验证按钮是否隐藏。

## 菜单配置

新增一个菜单：

```text
名称: 帮助中心文章
类型: 菜单
URL: shop/helpArticle
```

新增按钮权限：

```text
查看: shop:helpArticle:info,shop:helpArticle:page
新增: shop:helpArticle:save
修改: shop:helpArticle:update
删除: shop:helpArticle:delete
```

配置完成后，给目标角色分配菜单和按钮权限。

## 验证清单

- 登录管理后台后能看到菜单。
- 点击菜单能进入页面。
- 列表接口能返回分页数据。
- 新增、修改、删除按钮按权限显示。
- 无权限用户看不到按钮或访问接口失败。
- 新增数据写入 `tz_help_article`。

## 检查题-含答案

问题：为什么这个例子选“帮助中心文章”，而不是订单或支付？

答案：帮助文章是低风险后台 CRUD，能覆盖二开必学链路，但不会影响交易、库存、支付等高风险流程。

问题：后端接口写完后，为什么要先验证接口再写前端？

答案：这样能把问题分层。接口没通时先修后端；接口已通但页面失败时再查前端请求、表格字段或权限。

问题：菜单能看到但按钮看不到，最可能缺哪一步？

答案：角色没有分配按钮权限，或前端 `isAuth()` 的权限标识和 `tz_sys_menu.perms` 不一致。
